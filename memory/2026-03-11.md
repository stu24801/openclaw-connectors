## 14:30–14:58 — 504 Gateway Timeout 根本原因分析與串流修復

### 問題確認
- 景揚在 deepwiki.alex-stu24801.com 執行 wiki 生成時收到 `Error determining wiki structure: 504`
- 原因一：LLM Proxy 沒有實作 Streaming，整個回應要等完整生成後才送出，耗時 ~70 秒
- 原因二：Cloudflare Tunnel / Next.js 約 100 秒超時，LLM 回應慢時會先 504
- 原因三（當下）：Anthropic API 5h Unified quota exhausted（`anthropic-ratelimit-unified-5h-utilization: 1.0`），GitHub Copilot 也同時 429 quota_exceeded

### 修復內容
- 修改 `/opt/llm-proxy/main.py`：
  - 新增 `from fastapi.responses import JSONResponse, StreamingResponse`
  - 新增 `_stream_copilot()` — 直接串流 GitHub Copilot SSE 回應
  - 新增 `_stream_anthropic()` — 將 Anthropic SSE 轉換為 OpenAI chunk 格式後串流
  - 修改 `chat_completions()`：若 `body.get("stream", True)` 則走 `StreamingResponse`，避免卡住

### 驗證結果
- `curl /v1/chat/completions` 加 `stream=true` 現在立即回傳錯誤訊息而非卡住
- Gemini 端點 (`provider=google`) 仍正常運作（Google API key 有額度）

### 建議行動（景揚）
- 短期：在 DeepWiki 設定改用 Google Provider（Gemini 2.5 Flash 或 Pro）度過額度恢復期
- 長期：Anthropic 和 Copilot 額度恢復後可改回

## 18:48–19:04 香蕉簡報大綱生成修復

### 問題根因
- 香蕉簡報把 `api_key` 和 `api_base_url` 存在 **SQLite DB** (`/app/backend/instance/database.db`, `settings` table, id=1)，不是只讀 `.env`
- DB 裡儲存的是舊 token (`6n0vkCbk4zwytF_...`) 和舊 port (`9090`)
- Docker container 起來後直接從 DB 讀設定，`.env` 的修改無效
- 之前 `gen-banana-env.sh` 只更新 `.env`，所以無法解決

### 修復方式
- 升級 `gen-banana-env.sh`：同時更新 `.env` 和 DB
- 新增 `/opt/llm-proxy/update-banana-db.py`：透過 `docker exec` 在容器內執行 SQLite 更新
- DB 是 root 擁有，無法從 host 直接寫入，必須用 `docker exec`

### 關鍵檔案
- `/opt/llm-proxy/gen-banana-env.sh` — 啟動前同步 token 到 .env + DB
- `/opt/llm-proxy/update-banana-db.py` — docker exec 更新容器內 DB
- 香蕉簡報 DB: `/opt/banana-slides/backend/instance/database.db` (root 擁有)
- settings table: `id`, `api_key`, `api_base_url` 是關鍵欄位

### Token 輪換時操作
```bash
bash /opt/llm-proxy/gen-banana-env.sh
cd /opt/banana-slides && docker compose restart backend
```
